# Architecture Change: The Hybrid Crawler Engine

## 1. The Initial Approach: Fully based on HTTP Parsing
earlier, the crawler engine was built entirely around **Jsoup**, which acted as a fast, highly-concurrent HTML fetcher. Using a pool of 20 worker threads, the engine could rapidly download static HTML and parse the Document Object Model (DOM).

**The Advantage:** This was incredibly fast and required very little RAM, allowing for high-throughput crawling.

## 2. The Issue: The "JavaScript Wall" & Anti-Bot Systems
During testing against real-world targets a critical flaw in the pure-Jsoup approach popped. I encountered two major roadblocks:
1. **Single Page Applications:** Sites built with React/Angular returned nearly empty HTML shells. The actual content was rendered dynamically via JavaScript, which Jsoup cannot execute.
2. **Enterprise Firewalls :** Security-focused sites detected the HTTP requests as bot traffic and intercepted them with CAPTCHA pages (to be specific, mainly CloudFlare).

this resulted in crawler saving useless firewall warnings to the DB instead of the actual web content

## 3. Solution: A Hybrid Routing Architecture (suggested by Gemini & GPT)
To solve this without sacrificing the speed of Jsoup, I implemented a **Hybrid Crawler Architecture**.

The engine now acts as a smart router:
* **The Fast Lane (Jsoup):** Standard static sites are still processed instantly via lightweight HTTP requests.
* **The Heavy Lifter (Playwright):** If a page returns an abnormally small payload or contains known firewall keywords (e.g., "Just a moment"), the engine dynamically reroutes the URL to a headless browser implementation using Microsoft Playwright.

## 4. Playwright implementation
Integrating a full Chromium browser into a highly concurrent environment required strict resource management and anti-bot evasion techniques. I implemented the following mechanisms in the `fetchWithBrowser` method:

### A. Resource Protection (concurrency limiting)
launching a full browser is incredibly RAM-intensive. If all 20 threads hit a JavaScript wall simultaneously, the application would crash due to an `OutOfMemory` error.
* **Implementation:** I wrapped the Playwright invocation in a `Semaphore` (`browserLimiter.acquire()`). This forces the fast Jsoup threads to form a queue, ensuring only a safe, limited number of Chromium instances open at once, helping to not overwhelm the resources.

### B. Anti-Bot Evasion 
To successfully bypass Cloudflare and other Web Application Firewalls, the browser needs to behave like a human:
* **Visible Execution:** `setHeadless(false)` is used to bypass rudimentary headless-browser detection scripts.
* **Environment Spoofing:** The `Context` is injected with a standard Windows/Chrome User-Agent and a realistic viewport size to pass browser fingerprinting checks.
* **Behavioral Simulation:** The script waits for network idle states, applies a hard 6-second timeout to allow background CAPTCHAs to resolve, and simulates physical human interaction via mouse movements and scrolling (`page.mouse().wheel(0, 500)`).
* Cloudflare tracks our mouse. If a page loads and immediately clicks a link with zero mouse movement, it detects that. by moving the cursor slightly and scrolling down the page, we pass the "behavioral" bot check.

### C. Graceful Degradation & Cleanup
Every browser fetch is wrapped in a strict `try/finally` block. Regardless of whether the site successfully loads, times out, or permanently blocks the crawler, the `finally` block guarantees that the page is closed and the semaphore lock is released (`browserLimiter.release()`). This prevents memory leaks and "ghost tabs" from stacking up in the background.
